# CODE OS Security Model & Specifications

This document outlines the security architecture, boundaries, and data handling protocols of CODE OS as of today. 

CODE OS is designed as a **local-first** development environment. It executes on the developer's local machine, meaning the developer is responsible for the overall operating system security, but CODE OS implements internal boundaries to prevent unauthorized mutations by agents or extensions.

---

## 1. Workspace Trust Model (Restricted vs. Trusted Mode)

CODE OS features a strict workspace isolation model modeled on modern IDE security guidelines:
- **Restricted Mode**: When opening an untrusted workspace folder, CODE OS operates in a read-only sandboxed mode. File modifications, creation, deletion, git operations, terminal execution, and state-mutating MCP tools are blocked.
  - *Enforcement*: Restricted Mode is enforced server-side at the FastAPI route controller level (routes return `HTTP 403 Forbidden` if a workspace is restricted).
  - *Protected Surfaces*:
    - **File Operations**: Creation, writing, moving, renaming, duplicating, and deletion of files/folders are blocked server-side.
    - **Global Search-Replace**: Applying replacements (`/api/search/replace` with `apply=true`) is blocked.
    - **Git Mutations**: Staging and committing changes, pulling/pushing from/to remotes, and creating or switching branches are blocked. Read-only operations (getting status, diff, or history) remain active.
    - **PTY Terminals**: Creating new terminal sessions, executing command lines, and WebSocket terminal PTY handshakes are blocked server-side. Untrusted WebSocket connection attempts are immediately closed with an error.
    - **Model Context Protocol (MCP)**: Calls to custom servers are rejected. Default filesystem and git MCP servers are restricted to read-only tools (e.g. `read_file`, `list_directory`, `get_file_info`, `search_grep` on the filesystem server, and `git_status`, `git_diff`, `git_log` on the git server); any mutating commands (e.g. `write_file`, `git_commit`) are rejected.
  - *AI Boundary*: The edit-proposal pipeline blocks applying code modifications to files in restricted workspaces.
- **Trusted Mode (Full Trust)**: Allows complete file mutation, terminal command execution, and AI agent automation. Developers must explicitly trust the workspace via the trust consent banner in the UI.

## 2. Permission-Gated Execution

Autonomous agents and manual triggers cannot execute system-level operations without developer intervention:
- **Shell Commands**: All commands generated by AI agents (e.g., test runs, packages installation, scripts execution) must pass through the **PermissionGate** UI before executing.
- **File Writes & Proposals**: Edit proposals must be explicitly approved or rejected by the user inside the Diff Inspector before they are written to disk.
- **System Commands**: The system relies on explicit consent modal confirmations for destructive actions (e.g., deleting workspace files, clearing settings).

## 3. Secrets & API Key Encryption at Rest

All API keys for AI models (OpenAI Compatible base URLs, Anthropic, Groq, etc.) are protected to prevent leakage:
- **Encryption Algorithm**: Keys are encrypted at rest using **Fernet symmetric encryption** (from Python's `cryptography` library).
- **Key Storage**: Plaintext keys are never stored in the database or logged in the console. The encryption key is stored securely in the local user profile, and decryption happens solely in-memory when constructing request payloads to the model providers.

## 4. Local-First Data Model

CODE OS values developer privacy:
- All index data (AST parse trees, symbols, database tables) are stored locally in an embedded SQLite database (`~/.code-os/code-os.sqlite3`).
- **No Telemetry**: No tracking metrics, usage statistics, or intermediate files are uploaded to third-party endpoints.
- **Network Traffic**: The only outbound network requests are direct API calls to model provider endpoints that the user has explicitly configured with their own credentials.

## 5. Known Limitations & Risks

To remain credible, we disclose the following architectural limitations of the current local-first version:
- **No OS-Level Shell Sandboxing**: Once the developer approves a command through the permission gate, it runs in the host shell process. CODE OS does not execute commands in Docker containers or hypervisor-level sandboxes. 
  - *Hardening Countermeasure*: To mitigate risks, **Environment Sanitization** is enforced. Before any approved command or test execution is spawned by the backend, all credential-bearing environment variables (AI provider API keys, AWS keys, Git/npm tokens) are stripped from the child process's environment.
- **Electron Sandbox Disabled**: Electron's renderer process sandbox is currently disabled because the UI requires direct binding to `node-pty` for launching local shell instances in the Electron main/renderer process boundary.
- **No Third-Party Audit**: CODE OS is a personal developer tool and has not undergone a formal third-party penetration test or security audit.

## 6. Responsible Disclosure

If you discover a security vulnerability in CODE OS (such as a bypass of the restricted mode endpoint guards), please do not open a public issue. Instead, report it privately by contacting the project maintainer via email or through the private contact links on their profile page.
